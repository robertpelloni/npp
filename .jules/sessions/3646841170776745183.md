# Notepad++ Ultra-Project Memory & Architecture Summary

## 1. Project Vision & Ultimate Goals
The overarching goal is to transform Notepad++ from a tightly-coupled, Windows-only text editor into a **massive, cross-platform, multi-frontend "ultra-project."** 
Key transformations include:
*   **Total Win32 Decoupling:** Ripping out all `HWND`, `SendMessage`, and Windows API calls from the core business logic.
*   **The Go Port:** Systematically porting the entire backend and core architecture to Go, establishing it as the single source of truth for file state and configuration.
*   **Multiple Frontends (Submodules):** Using Go (or the abstracted C++ core) to drive multiple native UI bindings simultaneously, specifically `bobui` (Qt6), `btk` (Qt4), and `bobgui` (GTK).
*   **1:1 Geany Parity:** Building IDE-like features natively into the core, particularly a built-in terminal emulator and a compiler/build execution environment.

## 2. Core Architectural Decisions
*   **Event-Driven Architecture:** To break the Win32 `SendMessage` coupling, we are implementing a generic `EventBus` pattern. Both the transitional C++ core and the new Go backend use this to decouple Scintilla and the UI from the business logic.
*   **Abstracting Scintilla:** The legacy codebase communicates directly with Scintilla's internal window handle. We are routing all Scintilla interactions through an interface (`IScintillaBridge` in C++), which native UI wrappers will implement.
*   **Data Models First:** Data models like file buffering (`Buffer.cpp`) and configuration (`Parameters.cpp`) must be pure state, completely ignorant of the UI layer. We have successfully ported the `Buffer` and `BufferManager` concepts into Go (`go-port/pkg/core/buffer.go`).

## 3. Specific Feature Implementations
*   **Versioning Autosave:** Fulfilling the "Never lose a file" requirement, we implemented an immutable snapshotting system in Go (`pkg/autosave`). Instead of overwriting a temp file, it saves a ledger of all file states over time, deduplicating unchanged saves using SHA-256 hashes.
*   **Modern UI Demands:** The architecture is being prepped to support:
    *   **Vertical Tabs:** Modeled as adjustable-width columns rather than rotated 90-degree text.
    *   **Mixed Typography:** Structural support for rendering standard text with regular fonts and code with monospaced fonts seamlessly.
    *   **Plugin Integration:** Natively absorbing features from submodules like TextFX2 and Markdown viewing.

## 4. Operational & Agent Directives
*   **Documentation is Paramount:** Every architectural shift and version bump is meticulously documented across a suite of files (`VISION.md`, `ROADMAP.md`, `CHANGELOG.md`, `HANDOFF.md`, `SUBMODULES.md`, etc.).
*   **Strict Versioning:** Every commit and logical unit of work bumps the version (currently `v1.0.5`), tracked in the root `VERSION` file and `CHANGELOG.md`.
*   **Deep Commenting:** All touched and new code requires "deep comments" explicitly detailing the 'what', 'why', side effects, and design alternatives.
*   **Submodule Management:** External repositories are aggressively tracked as Git submodules with strict documentation of their current commit hashes and purposes.

## 5. Identified Anti-patterns in Legacy Code
*   **Styler Inheritance Breakage:** Hardcoding font styles directly into `stylers.model.xml` breaks Notepad++'s cascading style inheritance engine. This must be avoided in favor of the Style Configurator UI.
*   **Dialog Tying:** Logic is often embedded directly inside Dialog UI classes (`UserDefineDialog.cpp`, etc.), reading/writing directly to UI checkboxes instead of an underlying state model. This is the primary target for our decoupling efforts.

# Notepad++ Ultra-Project Memory & Architecture Summary

## 1. Project Vision & Ultimate Goals
The overarching goal is to transform Notepad++ from a tightly-coupled, Windows-only text editor into a **massive, cross-platform, multi-frontend "ultra-project."** 
Key transformations include:
*   **Total Win32 Decoupling:** Ripping out all `HWND`, `SendMessage`, and Windows API calls from the core business logic.
*   **The Go Port:** Systematically porting the entire backend and core architecture to Go, establishing it as the single source of truth for file state and configuration.
*   **Multiple Frontends (Submodules):** Using Go (or the abstracted C++ core) to drive multiple native UI bindings simultaneously, specifically `bobui` (Qt6), `btk` (Qt4), and `bobgui` (GTK).
*   **1:1 Geany Parity:** Building IDE-like features natively into the core, particularly a built-in terminal emulator and a compiler/build execution environment.

## 2. Core Architectural Decisions
*   **Event-Driven Architecture:** To break the Win32 `SendMessage` coupling, we are implementing a generic `EventBus` pattern. Both the transitional C++ core and the new Go backend use this to decouple Scintilla and the UI from the business logic.
*   **Abstracting Scintilla:** The legacy codebase communicates directly with Scintilla's internal window handle. We are routing all Scintilla interactions through an interface (`IScintillaBridge` in C++), which native UI wrappers will implement.
*   **Data Models First:** Data models like file buffering (`Buffer.cpp`) and configuration (`Parameters.cpp`) must be pure state, completely ignorant of the UI layer. We have successfully ported the `Buffer` and `BufferManager` concepts into Go (`go-port/pkg/core/buffer.go`).
*   **Centralized Configuration:** The legacy `Parameters.cpp` is being replaced by a unified Go `AppConfig` (`go-port/pkg/config/parameters.go`) that acts as the single source of truth for all UI toggles and editor behaviors.

## 3. Specific Feature Implementations
*   **Versioning Autosave:** Fulfilling the "Never lose a file" requirement, we implemented an immutable snapshotting system in Go (`pkg/autosave`). Instead of overwriting a temp file, it saves a ledger of all file states over time, deduplicating unchanged saves using SHA-256 hashes.
*   **Modern UI Demands:** The architecture natively supports:
    *   **Vertical Tabs:** Modeled as adjustable-width columns (`VerticalTabWidth` config) rather than rotated 90-degree text, managed in `pkg/workspace/layout.go`.
    *   **Mixed Typography:** Structural support for rendering standard text with regular fonts and code with monospaced fonts seamlessly (`MixFonts` toggle).
    *   **Intelligent Popups:** Configuration to disable autocomplete popups by default in non-code text files (`DisableAutocompleteInText` toggle).
    *   **ClearType & Themes:** Enforced/enhanced ClearType rendering and support for "super modern glass" aesthetic skins.
*   **Plugin Integration:** Natively absorbing features from submodules like TextFX2 and Markdown viewing.
*   **Geany Parity Foundations:** Built asynchronous build execution (`pkg/buildsys`) and pseudo-terminal interface scaffolding (`pkg/terminal`) in Go.

## 4. Operational & Agent Directives
*   **Documentation is Paramount:** Every architectural shift and version bump is meticulously documented across a suite of files (`VISION.md`, `ROADMAP.md`, `CHANGELOG.md`, `HANDOFF.md`, `SUBMODULES.md`, etc.).
*   **Strict Versioning:** Every commit and logical unit of work bumps the version (currently `v1.0.6`), tracked in the root `VERSION` file and `CHANGELOG.md`.
*   **Deep Commenting:** All touched and new code requires "deep comments" explicitly detailing the 'what', 'why', side effects, and design alternatives.
*   **Submodule Management:** External repositories are aggressively tracked as Git submodules with strict documentation of their current commit hashes and purposes.

## 5. Identified Anti-patterns in Legacy Code
*   **Styler Inheritance Breakage:** Hardcoding font styles directly into `stylers.model.xml` breaks Notepad++'s cascading style inheritance engine. This must be avoided in favor of the Style Configurator UI.
*   **Dialog Tying:** Logic is often embedded directly inside Dialog UI classes (`UserDefineDialog.cpp`, etc.), reading/writing directly to UI checkboxes instead of an underlying state model. This is the primary target for our decoupling efforts.
*   **Massive Switch Statements:** Legacy input handling relies on massive `switch` blocks (e.g., in `NppBigSwitch.cpp`). The Go port will aim to use a Command Pattern and the EventBus to map inputs to actions dynamically rather than structurally.

# Notepad++ Ultra-Project Memory & Architecture Summary

## 1. Project Vision & Ultimate Goals
The overarching goal is to transform Notepad++ from a tightly-coupled, Windows-only text editor into a **massive, cross-platform, multi-frontend "ultra-project."** 
Key transformations include:
*   **Total Win32 Decoupling:** Ripping out all `HWND`, `SendMessage`, and Windows API calls from the core business logic.
*   **The Go Port:** Systematically porting the entire backend and core architecture to Go, establishing it as the single source of truth for file state, configuration, and I/O.
*   **Multiple Frontends (Submodules):** Using Go (or the abstracted C++ core) to drive multiple native UI bindings simultaneously, specifically `bobui` (Qt6), `btk` (Qt4), and `bobgui` (GTK).
*   **1:1 Geany Parity:** Building IDE-like features natively into the core, particularly a built-in terminal emulator and a compiler/build execution environment.

## 2. Core Architectural Decisions & Patterns
*   **Event-Driven Architecture (`EventBus`):** To break the Win32 `SendMessage` coupling, we implemented a generic `EventBus` pattern in both the transitional C++ core and the new Go backend (`pkg/core/eventbus.go`). This decouples Scintilla and the UI from the underlying business logic.
*   **Abstracting Scintilla:** The legacy codebase communicates directly with Scintilla's internal window handle. We are routing all Scintilla interactions through an interface (`IScintillaBridge`), which the CGO layer (`pkg/bindings/scintilla_cgo.go`) and native UI wrappers implement.
*   **Data Models First:** Data models like file buffering (`Buffer.cpp`) and configuration (`Parameters.cpp`) must be pure state, completely ignorant of the UI layer. We successfully ported the `Buffer` and `BufferManager` concepts into Go (`go-port/pkg/core/buffer.go`).
*   **Centralized Configuration:** The legacy `Parameters.cpp` has been replaced by a unified Go `AppConfig` (`go-port/pkg/config/parameters.go`) acting as the single source of truth for all UI toggles and editor behaviors.
*   **Dynamic Command Routing (Command Pattern):** The massive, tightly-coupled `switch` statements from the legacy Win32 message loop (`NppBigSwitch.cpp`) were completely replaced by a Go-based `CommandManager` (`pkg/commands`). The Native C++ UI now triggers string-based actions (e.g., `File.New`) over a CGO boundary (`ExecuteCommandFromUI`), which safely mutates the Go state.
*   **Platform-Agnostic File I/O:** The legacy `NppIO.cpp` (which used Win32 `CreateFile`/`ReadFile`) has been replaced by a pure Go package (`pkg/io/fileio.go`) utilizing standard library file operations, ensuring cross-platform seamlessness.
*   **Agnostic Plugin Manager:** A unified interface for loading extensions (`pkg/plugins/manager.go`) abstracts away the OS-specific DLL loading mechanics that plagued the old architecture, preparing for native TextFX2 integration.

## 3. Specific Feature Implementations
*   **Versioning Autosave:** Fulfilling the "Never lose a file" requirement, we implemented an immutable snapshotting system in Go (`pkg/autosave`). Instead of overwriting a temp file, it saves a ledger of all file states over time, deduplicating unchanged saves using SHA-256 hashes.
*   **Modern UI Demands:** The architecture natively supports:
    *   **Vertical Tabs:** Modeled as a core Layout state (`pkg/workspace/layout.go`) with adjustable-width columns (`VerticalTabWidth`), discarding the legacy rotated 90-degree text approach.
    *   **Mixed Typography:** Structural support for rendering standard text with regular fonts and code with monospaced fonts seamlessly (`MixFonts` toggle).
    *   **Intelligent Popups:** Configuration to disable autocomplete popups by default in non-code text files (`DisableAutocompleteInText` toggle).
    *   **ClearType & Themes:** Enforced/enhanced ClearType rendering and support for "super modern glass" aesthetic skins (`SkinTheme` config).
*   **Geany Parity Foundations:** Built asynchronous build execution pipelines (`pkg/buildsys`) and cross-platform pseudo-terminal interface scaffolding (`pkg/terminal`) directly into the Go core.

## 4. Operational & Agent Directives
*   **Documentation is Paramount:** Every architectural shift and version bump is meticulously documented across a suite of files (`VISION.md`, `ROADMAP.md`, `CHANGELOG.md`, `HANDOFF.md`, `SUBMODULES.md`, `PHASE_ANALYSIS.md`, etc.).
*   **Strict Versioning:** Every commit and logical unit of work bumps the version (currently at `v1.0.8`), tracked in the root `VERSION` file and `CHANGELOG.md`. Version bumps are explicitly referenced in git commit messages.
*   **Deep Commenting:** All touched and new code requires "deep comments" explicitly detailing the 'what', 'why', side effects, and design alternatives.
*   **Submodule Management:** External repositories are aggressively tracked as Git submodules with strict documentation of their current commit hashes and purposes in `SUBMODULES.md`.

## 5. Identified Anti-patterns in Legacy Code
*   **Styler Inheritance Breakage:** Hardcoding font styles directly into `stylers.model.xml` breaks Notepad++'s cascading style inheritance engine. This must be avoided in favor of the Style Configurator UI.
*   **Dialog Tying:** Logic is often embedded directly inside Dialog UI classes (`UserDefineDialog.cpp`, etc.), reading/writing directly to UI checkboxes instead of an underlying state model. Our Go configuration and command architecture solves this.
*   **The Big Switch:** Routing logic via huge `switch` blocks checking Windows Message IDs (`WM_COMMAND`) is rigid and unportable. The Go Command Pattern resolves this issue natively.
